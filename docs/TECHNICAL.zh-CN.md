# Memcode 技术文档

## 概述

本文档解释 memcode 的核心实现原理、为什么使用 MCP，以及系统内部的工作机制。

## 问题背景

当 Claude Code 通过文件系统读取代码文件时，需要重复执行磁盘 I/O 操作。对于大型代码库，这可能会很慢且效率低下。

**解决方案**：将所有代码一次性加载到内存中，然后直接从内存提供文件内容，而不是每次都从磁盘读取。

## 系统架构

### 核心组件

```
┌─────────────────────────────────────────────────────────────────┐
│                        Claude Code                               │
│                     (VSCode 扩展)                                │
└──────────────────────────┬──────────────────────────────────────┘
                          │ stdio (标准输入/输出)
                          │ JSON-RPC 2.0
┌──────────────────────────▼──────────────────────────────────────┐
│                          memcode                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    MCP 协议层                              │   │
│  │              (JSON-RPC 请求/响应处理)                      │   │
│  └────────────────────────┬────────────────────────────────┘   │
│                          │                                      │
│  ┌───────────────────────▼────────────────────────────────┐   │
│  │                   内存文件缓存                             │   │
│  │   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐    │   │
│  │   │ file1   │ │ file2   │ │ file3   │ │ fileN   │    │   │
│  │   │ .go     │ │ .ts     │ │ .py     │ │ .md     │    │   │
│  │   └─────────┘ └─────────┘ └─────────┘ └─────────┘    │   │
│  └────────────────────────┬────────────────────────────────┘   │
│                          │                                      │
│  ┌───────────────────────▼────────────────────────────────┐   │
│  │                  文件监视器 (可选)                         │   │
│  │              (fsnotify / chokidar)                       │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 通信流程

```
1. Claude Code 需要读取文件
      │
      ▼
2. 通过 stdin 发送 JSON-RPC 请求
   {
     "jsonrpc": "2.0",
     "id": 1,
     "method": "tools/call",
     "params": {
       "name": "memory_code_read",
       "arguments": { "path": "src/main.go" }
     }
   }
      │
      ▼
3. memcode 接收请求
      │
      ▼
4. 从内存缓存中查找文件内容
      │
      ▼
5. 通过 stdout 返回结果
   {
     "jsonrpc": "2.0",
     "id": 1,
     "result": {
       "content": "package main\n..."
     }
   }
```

## 为什么要用 MCP？

### MCP 是必须的吗？

**不是。** MCP 并不是必须的。核心功能完全可以不用 MCP 实现：

```
# 不用 MCP - 自定义协议
echo '{"path": "src/main.go"}' | ./memcode --protocol custom
```

### 那为什么还要用 MCP？

| 对比项 | 不用 MCP | 用 MCP |
|--------|---------|--------|
| 协议设计 | 需要自己设计 | 行业标准 |
| 集成方式 | 手动实现 | 内置支持 |
| 配置方式 | 复杂 | 简单 JSON |
| 生态 | 封闭 | 互操作性强 |
| 文档 | 从零开始 | 遵循惯例 |

**简单说**：MCP 提供了标准化的方式：
1. 定义可用的工具
2. 处理请求/响应格式
3. 无缝集成到 Claude Code

## 可用工具

Memcode 暴露了以下 MCP 工具：

### 1. memory_code_read

从内存缓存读取文件内容。

```typescript
// 参数
{
  path: string  // 相对或绝对文件路径
}

// 响应
{
  content: string,  // 文件内容
  size: number      // 文件大小（字节）
}
```

### 2. memory_code_grep

在缓存的文件中搜索模式。

```typescript
// 参数
{
  pattern: string,   // 正则表达式模式
  path?: string      // 可选：指定搜索目录
}

// 响应
{
  matches: Array<{
    file: string,
    line: number,
    content: string
  }>
}
```

### 3. memory_code_glob

查找匹配 glob 模式的文件。

```typescript
// 参数
{
  pattern: string  // glob 模式 (例如: "src/**/*.ts")
}

// 响应
{
  files: string[]  // 匹配的文件路径列表
}
```

### 4. memory_code_stats

获取缓存统计信息。

```typescript
// 响应
{
  totalFiles: number,
  totalSize: number,
  lastReload: string  // ISO 时间戳
}
```

## 实现细节

### Go 版本

**文件加载**：递归遍历目录，将文件读入 `map[string]string`

**文件监视**：使用 `fsnotify` 库监控文件变化

**MCP 协议**：基于 stdio 的自定义 JSON-RPC 2.0 实现

### npm/TypeScript 版本

**文件加载**：使用 `fast-glob` 发现文件

**文件监视**：使用 `chokidar` 实现跨平台文件监视

**MCP 协议**：使用 `@modelcontextprotocol/server` SDK

## 性能特性

| 操作 | 磁盘 I/O | 内存 |
|-----------|----------|--------|
| 读取文件 | ~1-10ms | ~0.01ms |
| 搜索 | ~10-100ms | ~1-10ms |
| glob | ~5-50ms | ~0.1-1ms |

**典型提升**：重复操作快 10-100 倍

## 内存占用

内存存储通常比原始文件大 **1.5-3 倍**。

### 开销来源

| 来源 | 说明 |
|--------|-------------|
| **string 副本** | Go 的 string 是 immutable，读取时会复制一次 |
| **Map 键** | 文件路径作为 key 存储 |
| **Map 负载因子** | Go map 约有 15% 空闲空间 |
| **string 头部** | 每个 string 有 16-32 字节头部开销 |

### 示例

假设一个 10MB 的代码库：

| 项目 | 大小 |
|------|------|
| 源文件 | 10 MB |
| 内存占用（估算） | 15-25 MB |
| 增量开销 | 5-15 MB |

### 权衡

```
读取文件 1000 次:
  磁盘 I/O: 1000 × ~5ms = 5 秒
  内存读取: 1000 × ~0.01ms = 0.01 秒
```

对于代码编辑场景，额外的内存开销换取 100-500 倍的读取速度是值得的。

### 优化建议

如果内存是瓶颈，可以：

1. **减少包含的文件类型** - 用 `--include` 只加载核心代码
2. **排除大目录** - 用 `--exclude` 排除 test、vendor 等
3. **未来版本** - 可考虑 LRU 淘汰策略，只保留最近访问的文件

## 配置

### 命令行选项

| 选项 | 默认值 | 描述 |
|--------|---------|-------------|
| `--path` | （必填） | 代码目录路径 |
| `--exclude` | .git,node_modules,__pycache__,venv | 要排除的目录 |
| `--include` | .js,.ts,.jsx,.tsx,.py,.go,.java,.c,.cpp,.h,.rs,.md,.json | 要包含的文件扩展名 |
| `--watch` | true | 启用文件监视 |
| `--no-watch` | - | 禁用文件监视 |
| `--debounce` | 500 | 防抖延迟（毫秒） |

### 示例

```bash
# 将 /path/to/project 加载到内存
./memcode --path /path/to/project

# 排除额外目录
./memcode --path /path/to/project --exclude .git,node_modules,vendor

# 只包含特定扩展名
./memcode --path /path/to/project --include .go,.ts,.py
```

## 局限性

1. **首次加载**：首次加载时间与代码库大小成正比
2. **同步延迟**：文件监视器在更新反映到缓存之前可能有轻微延迟
3. **二进制文件**：不建议用于大型二进制文件
